# Submission Plan

## Objective

Submit a working, clean, believable prototype with enough clarity that judges can run it and understand the value quickly.

## Required Submission Contents

- source code
- run instructions
- preprocessing instructions
- assumptions and limitations
- demo assets

## Required Files

- `README.md`
- `package.json`
- `requirements.txt`
- `.env.example`
- `scripts/preprocess_parking_data.py`
- app source
- `docs/`

## README Structure

1. Problem statement selected
2. Solution summary
3. Screens included
4. Tech stack
5. Dataset notes
6. Local run steps
7. Preprocessing step
8. Known assumptions
9. Demo flow

## Demo Assets

- 5 to 8 screenshots
- 2 to 3 minute demo video
- optional architecture diagram

## Packaging Rules

- Keep raw dataset outside the main zip if size becomes a problem
- If needed, include a sample processed subset plus instructions for full preprocessing
- Do not require cloud services for core demo

## Judge-Facing Narrative

Position LaneGuard as:

- operational
- explainable
- high-impact
- grounded in real parking enforcement data
- capable of suggesting data-driven interventions

## Final Quality Checklist

- app runs from fresh install
- map and filters work
- at least one hotspot details flow is polished
- planner view works
- daily brief works
- recommendation layer works and is explainable
- README is accurate
- assumptions are explicit
